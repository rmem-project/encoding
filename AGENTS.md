# Правила репозиторію

## Структура проєкту та модулів

Репозиторій зараз перебуває на етапі специфікації. Основний артефакт — `docs/SPEC.md`, де описано цільову архітектуру TypeScript-бібліотеки `@rmem/md-parser`. Папка `docs/` використовується як пам'ять проєкту: тримайте там архітектурні рішення, правила сумісності, поведінку парсера та інші знання, потрібні для майбутньої роботи.

Коли з'явиться код, використовуйте типову структуру TypeScript-бібліотеки:

- `src/` — parser, runtime, portals, renderer, signatures та інтеграція encoding service.
- `tests/` — unit- та integration-тести.
- `fixtures/` — Markdown-приклади, edge cases для кодувань і snapshots рендерера.
- `docs/` — архітектурні нотатки та специфікації публічної поведінки.

## Пам'ять проєкту

`docs/index.md` є реєстром усіх документів пам'яті проєкту. Він має містити дерево записів у форматі `шлях/назва_файлу` — короткий опис документа.

Перед виконанням завдань досліджуйте пам'ять проєкту на предмет корисних знань. Для цього спочатку відкривайте `docs/index.md`, а потім релевантні документи з індексу.

Під час додавання, видалення або значної зміни документа в `docs/`, яка змінює його короткий опис, завжди актуалізуйте `docs/index.md` у тому самому наборі змін.

Організовуйте документи в папках так, щоб їх було зручно знаходити: для простих секцій достатньо одного рівня, для складніших допускається два рівні вкладеності.

`docs/plan/progress.md` є документом прогресу реалізації. Після того як користувач зафіксує виконання задачі, потрібно одразу актуалізувати цей документ: позначити задачу виконаною, за потреби додати коротку примітку та оновити пов'язані записи в `docs/index.md`, якщо змінився склад або опис документів.

## Стиль коду та іменування

Цільова мова реалізації — TypeScript. Публічні контракти мають бути явно типізованими. Використовуйте PascalCase для класів, інтерфейсів і типів, camelCase для функцій та змінних, kebab-case для package-level файлів, якщо це доречно.

Архітектура має бути production-first: реалізуйте стабільні абстракції, які відповідають `docs/SPEC.md`, без тимчасових моделей, які доведеться викидати.

## Мовні правила

Розмову з користувачем потрібно вести українською мовою. Документацію та коментарі в коді також потрібно писати українською мовою.

Усі повідомлення, які генеруються кодом для виводу користувачу, логів, CLI, помилок або diagnostics, мають бути англійською мовою.

## Правила тестування

Коли з'являться тести, перевіряйте поведінку перед внутрішньою реалізацією. 
Назви тестів мають описувати поведінку. Тримайте fixtures малими та сфокусованими.

## Коміти та Pull Request

З цієї директорії Git-історія зараз недоступна, тому наявний стиль комітів визначити неможливо. До появи власної конвенції використовуйте короткі Conventional Commit-повідомлення, наприклад `docs: refine parser architecture` або `test: add encoding fixtures`.

Pull Request має містити короткий опис, посилання на пов'язані issues або розділи специфікації, підтвердження тестами та нотатки про сумісність або архітектурні компроміси.

## Інструкції для агентів

Не зрізайте кути. Якщо поведінка неочевидна, уточнюйте замість припущень. Виправляйте першопричини, а не симптоми, і обґрунтовуйте архітектурні рішення логікою, прив'язаною до специфікації.
